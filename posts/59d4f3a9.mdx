## 笔记整理

<p>并行化是 Sub-agents 系统最核心的价值所在。当多个独立任务可以同时推进时，合理的拆分策略能 将工作效率提升数倍。本节介绍四种经过实践验证的拆分模式，以及并行与顺序执行的决策框架和结 果聚合方法。</p>

<p>四种任务拆分策略 策略一：领域拆分（Domain Split） 最常用的拆分方式。每个 Agent 拥有独立的文件域，零文件重叠，从根本上消除写冲突。</p>

<p>src/components/ ⊕ src/api/ ⊕ src/db/ + migrations/ ⊕ 领域拆分的前提是明确的目录边界。在开始前，先用 tree 或 Glob 工具绘制目录结构，确保每个 Agent 的文 件域没有交集。</p>

<p>策略二：视角拆分（Perspective Split） 适合分析、调研类任务。同一代码库或主题，从不同专业角度并行审视，发现单一视角遗漏的问题。</p>

<p>安全漏洞 / 注入 / XSS | ⊕</p>

<p>N+1 查询 / 性能瓶颈 | ⊕</p>

<p>测试覆盖率 / 边界条件 视角拆分天然无文件冲突——各 Agent 只读不写。结果汇聚后，主会话按优先级整合多个视角的发现。</p>

<p>策略三：假设拆分（Hypothesis Split） 专为调试场景设计。对同一个 bug，并行验证多个互相竞争的根因假设，并配置一个&quot;魔鬼代言人&quot;Agent 专门 反驳其他假设。</p>

<p>JWT Token 过期？ | ⊕</p>

<p>并发竞争条件？ | ⊕</p>

<p>数据库连接泄漏？ | ⊕</p>

<p>反驳前三者 经过假设竞争幸存下来的理论，往往就是最可能的根本原因。这种模式在面对线上事故时尤为有效。</p>

<p>策略四：流水线拆分（Pipeline Split） 当任务存在严格的顺序依赖时，将各阶段转化为专属 Agent，前一阶段的输出即是下一阶段的输入。</p>

<p>最终输出 流水线拆分虽然是顺序执行，但每个阶段有专属的 context 空间和工具权限，确保每个 Agent 聚焦自己的职 责，不被前后阶段的&quot;噪声&quot;干扰。</p>

<p>并行 vs 顺序决策矩阵 在决定是否并行之前，用这张决策矩阵快速评估：</p>

<p>判断维度 | 并行执行 | 顺序执行</p>

<p>任务相互依赖 | 无依赖，任务独立 | B 需要 A 的输出</p>

<p>文件冲突风险 | 无重叠 / 使用 worktree | 操作同一文件</p>

<p>任务数量 | 3 个以上独立任务 | 少于 3 个任务</p>

<p>状态共享 | 不共享状态 | 共享状态 / 数据库</p>

<p>Token 效率 | 独立调研类任务 | 有强依赖链的任务</p>

<p>核心建议 并行的代价是协调开销。当任务粒度过细时，子代理的启动、context 传递、结果汇聚的开销可能超过并行带 来的收益。经验法则：单个任务预计耗时小于 2 分钟时，不值得并行化。</p>

<p>结果聚合的四种方法 方法一：文件聚合 每个 Agent 将结果写入独立文件，主会话最后读取并综合。适合内容较多、需要留存中间过程的场景。</p>

<p># Agent A、B、C 分别输出到独立文件 # 主会话指令 读取这三份报告，整合为一份按优先级排序的问题清单 方法二：直接返回综合 Sub-agents 模式下，各 Agent 完成后结果自动返回主会话 context，Claude 原生汇聚。适合结果简短、需要 立即综合的场景。</p>

<p>分别用三个子代理调研 Redis、Memcached、Hazelcast 的集群方案， 然后综合三份调研，给出选型建议 方法三：协商汇聚（Agent Teams） 在 Agent Teams 模式中，Teammates 直接对话、互相挑战，Lead 监控讨论并最终综合共识。适合需要&quot;辩 论&quot;才能收敛的复杂问题。</p>

<p>方法四：结构化输出 要求每个 Agent 以固定格式输出，便于程序化处理或自动合并。适合批量处理、CI 流水线集成的场景。</p>
