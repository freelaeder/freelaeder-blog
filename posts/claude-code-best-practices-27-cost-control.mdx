## 笔记整理

<p>token 消耗直接转化为账单成本，也影响会话质量——上下文越满，Claude 的表现越差。本节提供从 上下文管理到 Hook 预处理的系统性成本控制策略，附具体数据对比。</p>

<p>成本结构认知 在控制成本之前，先理解成本从哪里来：</p>

<p>使用场景 | 每日成本（Sonnet） | 说明</p>

<p>轻度使用（探索、问答） | $1–3 | 上下文窗口小，交互少</p>

<p>中度使用（日常编码） | $6 | 每开发者平均水平</p>

<p>重度使用（大型重构） | $12–20 | 90% 用户的上限</p>

<p>月均估算 | $100–200/人 | 使用 Sonnet 4.6</p>

<p>成本与上下文大小近似线性增长。一个充满大量文件内容的上下文窗口，每轮交互的成本可能是空白会话的 10– 20 倍。</p>

<p>Token 消耗分析工具 /cost # 查看当前 session 的 token 使用情况 /stats # 查看使用模式统计（Claude Max/Pro 订阅者） /context # 查看当前占用上下文空间的内容明细 在开始大型任务前，先用 `/context` 确认上下文空间，避免任务到一半因上下文满而中断。</p>

<p>/compact 策略 当上下文快满时，`/compact` 会生成一份压缩摘要替换原有对话历史。关键是：可以指定压缩时优先保留什么 信息。</p>

<p>/compact 重点保留 API 改动和已通过的测试命令 在 CLAUDE.md 中预定义压缩行为，避免每次手动指定：</p>

<p># 压缩指令 使用 /compact 时，请重点保留：</p>

<p>- 已修改的文件列表（含修改原因） - 关键测试命令和运行方式 - 已确认的架构决策 - 当前正在处理的 bug 描述和复现步骤 核心建议 合理使用 /compact 的时机：当 /cost 显示 token 使用量超过 60% 时，主动压缩。不要等到满了再压缩，那时 候摘要质量会下降。</p>

<p>模型选择策略 不同任务对模型能力的需求差异巨大，一律使用 Sonnet 是成本浪费，一律使用 Opus 是过度开销：</p>

<p>任务类型 | 推荐模型 | 成本对比 | 理由</p>

<p>日常编码、文件操作（约80%任务） Sonnet 基准性价比最高，能力足够 复杂架构决策、多步深度推理 Opus 约5x 推理深度显著更强 简单子代理、文件扫描、格式转换 Haiku 约0.1x 成本极低，速度快 在 Skill 中为子代理指定轻量模型：</p>

<p>--- description: 扫描文件提取关键信息，适合大批量处理 --- 扫描指定文件并提取关键信息... Extended Thinking（扩展思考）默认开启，能显著提升复杂任务质量，但代价是每次请求可能消耗数万额外 token（按输出 token 计费，单价更高）。</p>

<p>调整思考深度：</p>

<p>/effort low # 禁用扩展思考（简单任务） /effort medium # 默认（平衡） /effort high # 最大推理深度（复杂架构决策） 单次触发超深度推理（不改变全局配置）：</p>

<p>ultrathink 这个分布式事务的边界情况应该如何处理？</p>

<p>`ultrathink` 关键词只对当前这轮生效，是触发深度分析的轻量方式。</p>

<p>环境变量控制（CI/CD 或脚本场景）：</p>

<p>export MAX_THINKING_TOKENS=0 # 完全禁用扩展思考</p>

<p>场景 | 推荐设置 | 预期节省</p>

<p>日常代码改动、格式修复 /effort low 或 MAX_THINKING_TOKENS=0 30–50%</p>

<p>中型功能开发 | /effort medium（默认） | —</p>

<p>系统设计、算法选型 /effort high 或 ultrathink 质量提升 &gt; 成本增加 Hook 预处理数据 Hook 最重要但最被低估的用途之一：在数据进入 Claude 上下文之前过滤和压缩它。一个 100MB 的日志文 件，Claude 能读取到的往往只有最后几千行，其余全是浪费。</p>

<p>通过 PreToolUse Hook，在测试命令执行时只保留失败信息：</p>

<p># filter-test-output.sh：只显示测试失败，节省大量 token # 将命令替换为过滤版本：只输出 FAIL/ERROR 的前后 5 行 echo &quot;{\&quot;hookSpecificOutput\&quot;:{\&quot;permissionDecision\&quot;:\&quot;allow\&quot;,\&quot;updatedInput\&quot;:{\&quot;command 配置到 settings.json：</p>

<p>{ { { } ] } ] } } 实测数据：对于包含 10,000 行输出的测试套件，过滤后通常只剩 50–200 行关键错误信息，token 消耗减少 90% 以上。</p>

<p>子代理隔离高 Token 操作 运行测试、获取文档、处理日志文件等操作会消耗大量上下文。将这些委托给子代理，详细输出留在子代理的 独立上下文中，只有摘要返回主会话：</p>

<p>使用子代理运行完整测试套件并整理所有失败的测试， 只向我汇报：</p>

<p>1. 失败测试列表（文件名 + 测试名） 2. 每个失败的一句话原因摘要 不要输出完整的测试日志。</p>

<p>精确 Prompt 避免无效扫描 不推荐 推荐 「改进这个代码库」— 触发 Claude 读取大量文件，消 「为 src/auth/login.ts 中的 login() 函数添加输入校 耗数千 token，但改动范围模糊。</p>

<p>验，校验规则参考 src/validation/schemas.ts 中现有 的模式」— 精确指向两个文件，token 消耗极低。</p>

<p>MCP 服务器瘦身 每个启用的 MCP 服务器都会向上下文添加工具定义，即使你本次任务完全不会用到这些工具。</p>

<p>/mcp # 管理并禁用不需要的 MCP 服务器 有 CLI 工具时优先用 CLI（如 `gh`、`aws`、`gcloud`）替代对应的 MCP 服务器——CLI 不占上下文空间， MCP 工具定义随会话持续占用。</p>
