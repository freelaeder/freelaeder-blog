## 笔记整理

<p>Claude Code 不只是代码工具，它在技术写作、翻译、批量内容生成方面同样强大。本节介绍如何通 过 Skill 化写作规范、术语控制和流水线设计，让内容创作质量稳定、可复用。</p>

<p>写作规范 Skill 化 写作规范写在 CLAUDE.md 中有一个问题：不管你在写代码还是写文档，这些规范每次都会被加载，浪费 token。更好的做法是将写作规范封装成 Skill，只在需要时调用。</p>

<p>创建博客写作 Skill（`.claude/skills/blog-writer/SKILL.md`）：</p>

<p>--- description: 技术博客文章写作，保持一致的品牌风格 --- # 写作规范 - 标题：简洁有力，包含关键词，不超过 20 个字 - 开头：以问题或具体场景切入，不要以&quot;在今天的...&quot;开头 - 代码示例：所有代码必须可实际运行，附注释说明关键行 - 禁用词：避免使用&quot;非常&quot;、&quot;很&quot;、&quot;总之&quot;等虚词 - 结尾：以行动号召或延伸阅读建议结束 # 工作流 1. 分析读者画像和目标，确认关键信息传达优先级 2. 创建文章大纲供确认，再开始写正文 3. 先写代码示例，再围绕代码写文字说明 4. 自检：是否有无法证实的声明？是否有术语不一致？</p>

<p>使用时只需：</p>

<p>/blog-writer 写一篇关于 React Server Components 的技术博客， 目标读者是有 2-3 年 React 经验的前端开发者。</p>

<p>核心建议 Skill 的核心价值：将重复使用的工作流规范从&quot;每次加载&quot;变成&quot;按需加载&quot;。CLAUDE.md 目标控制在 500 行以 内，超出部分移到 Skill。</p>

<p>翻译中的术语控制 技术翻译最大的挑战是术语一致性。同一个词在不同段落用不同的中文翻译，会让读者感到困惑，也让后续的 机器搜索失效。解决方案是在翻译 Prompt 中预定义术语表：</p>

<p>翻译这段英文技术文档为中文，要求：</p>

<p>术语对照（严格遵守，不可变动）：</p>

<p>- middleware → 中间件 - token → token（不翻译） - session → 会话 - hook → 钩子 - callback → 回调函数 - repository → 仓库 翻译规范：</p>

<p>1. 上述术语之外的技术名词保留英文原词 2. 使用惯用的中文技术写作风格，而不是逐字翻译 3. 代码注释同步翻译，代码本身不翻译 4. 保持原文的标题层级结构 对于长期项目，将术语表放入 CLAUDE.md 的&quot;翻译规范&quot;章节，每次翻译任务自动继承，避免重复定义。</p>

<p>翻译质量保证 翻译完成后，让 Claude 自我校验是一个高性价比的质量控制手段：</p>

<p>检查 @docs/zh/api-reference.md 的翻译质量：</p>

<p>1. 是否有术语不一致的地方（参照术语表）？</p>

<p>2. 有没有遗漏的段落（与 @docs/en/api-reference.md 逐段对比）？</p>

<p>3. 技术描述是否准确？</p>

<p>4. 代码示例的注释是否已翻译？</p>

<p>输出一份问题清单，附上具体位置（行号）和建议修改方案。</p>

<p>批量内容生成策略 当需要生成大量结构相似的内容时（如为 50 个 API 端点写文档注释，为 100 个产品写描述文案），有两种主要 策略：</p>

<p>策略一：非交互模式并行处理（首选） # 并行处理多个文件，每个 Claude 调用独立上下文 claude -p &quot;为 $file 中所有没有文档的函数添加 JSDoc 注释， 参考 UserService.js 中现有注释的风格。只输出 OK 或 FAIL。&quot; \ 这种方式每个文件独立上下文，不会相互污染，且可以充分利用并行性。</p>

<p>策略二：子代理并行（适合需要跨文件上下文的任务） 使用 3 个子代理并行处理文档生成任务：</p>

<p>- 子代理 A：处理 src/auth/ 模块的文档 - 子代理 B：处理 src/payment/ 模块的文档 - 子代理 C：处理 src/notification/ 模块的文档 每个子代理完成后汇报已处理的函数数量和跳过的函数（及原因）。</p>

<p>不推荐 推荐 在单个 session 中顺序生成 50 个文档，上下文快速膨 使用非交互模式并行启动多个独立 Claude 进程，每个 胀，后期质量下降，且一旦出错需要重头来过。</p>

<p>处理少量文件，质量稳定，单个失败不影响其他任务。</p>

<p>利用图像辅助内容创作 Claude Code 支持直接粘贴截图作为输入，这在内容创作场景中有实用价值：</p>

<p>[粘贴产品截图] 根据这个截图，帮我写一段功能介绍文案， 面向非技术产品经理读者， 重点突出用户价值而不是技术实现， 控制在 150 字以内。</p>

<p>对于 UI 设计稿的文案填写，截图比文字描述更精确，也能避免 Claude 对界面布局的误解。</p>

<p>技术文档的识别与生成 识别缺少文档的代码，再批量生成，是内容创作中最常见的任务之一：</p>

<p># 第一步：识别 找到 auth 模块中没有 JSDoc 注释的函数，列出函数名和所在文件。</p>

<p># 第二步：批量生成（引用现有风格） 为上述函数添加 JSDoc 注释， 参考 UserService.js 中现有注释的风格和详细程度。</p>

<p>每个注释需包含：功能描述、参数说明、返回值、可能的异常情况。</p>
