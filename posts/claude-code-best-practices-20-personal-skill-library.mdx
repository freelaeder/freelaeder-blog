## 本节重点

- §20.1 从零创建 Skill 的完整流程

## 笔记整理

<p>了解 Skills 系统的机制只是第一步，真正的价值在于将其转化为属于你自己的、可不断积累的技能 库。本节提供从零创建 Skill 的完整流程、用户级与项目级的决策框架，以及三个完整的实战案例，帮 助你将重复性工作一次性&quot;固化&quot;为 Skill。</p>

<h2>§20.1 从零创建 Skill 的完整流程</h2>

<p>1 | 识别值得&quot;固化&quot;的工作模式</p>

<p>当你发现自己第三次重复相同的提示词时，就是创建 Skill 的时机。理想的 Skill 候选：频率高（每周至少 一次）、步骤固定（流程不随项目变化）、易出错（需要提醒和检查清单）。</p>

<p>2 | 选择用户级还是项目级</p>

<p>遵循以下决策树：</p>

<p>这个 Skill 是否：</p>

<pre><code>├── 只对这个项目有意义？→ 项目级（.claude/skills/）</code></pre>

<pre><code>├── 包含项目特定的命令/路径？→ 项目级</code></pre>

<pre><code>├── 需要团队其他成员使用？→ 项目级（提交到 git）</code></pre>

<pre><code>└── 是个人工作习惯？→ 用户级（~/.claude/skills/）</code></pre>

<p>3 | 创建目录和 SKILL.md</p>

<p>以创建一个用户级 Skill 为例：</p>

<p>然后编写 SKILL.md，从最简模板开始：</p>

<p>--- description: 简洁描述功能和触发场景 --- ## 任务 执行以下步骤：</p>

<p>1. 步骤一 2. 步骤二 ## 输出格式 [描述期望的输出格式]</p>

<p>4 | 写好 description（最重要的一步）</p>

<p>description 决定 Skill 能否被 Claude 自动识别。写作要点：</p>

<p>包含用户自然会说出的关键词和短语 明确列举触发场景（&quot;当用户说…时&quot;） 与其他 Skill 区分，避免混淆触发 略微积极主动，对抗 Claude 的&quot;欠触发&quot;倾向</p>

<p>5 | 验证和调试</p>

<p>创建后的验证步骤：</p>

<p># 列出所有可用 Skills，确认新 Skill 出现 # 直接手动调用，验证 Skill 本身可用 # 检查上下文，确认 description 未被截断 # 用自然语言触发，验证自动激活（如果需要） &quot;帮我[与 description 相符的任务]&quot;</p>

<p>6 | 迭代完善</p>

<p>Skill 的最佳状态来自实际使用的打磨。常见改进方向：description 更精准的触发词、SKILL.md 正文的 步骤更细化、添加 examples/ 目录展示期望输出、将常用参考放入 references/ 按需加载。</p>

<h2>§20.2 SKILL.md 写作最佳实践</h2>

<p>三条写作铁律：</p>

<p>1. 用祈使句：「读取文件」而非「应该读取文件」，Claude 对直接指令响应更好 2. 解释&quot;为什么&quot;：告知 Claude 某步骤的原因，比写 MUST/ALWAYS/NEVER 更有效 3. 正文控制在 500 行内：大型参考文档放 references/，在 SKILL.md 中明确指引</p>

<h2>§20.3 实战案例一：/morning-review — 晨间代码回顾</h2>

<p>背景：每天早上需要回顾昨天未完成的任务和代码变更，确认今天的开始状态。</p>

<p>--- description: 晨间代码状态回顾，汇总未完成工作和今日计划 --- ## 工作状态扫描 当前分支情况：</p>

<p>- 本地分支列表：!`git branch -v` - 未合并的变更：!`git stash list` - 未完成的工作区：!`git status --short` ## 生成晨间报告 根据以上信息，生成一份简洁的晨间状态报告：</p>

<p>**代码状态**：</p>

<p>- 当前分支：[当前在哪个分支，有无未提交变更] - 暂存的工作：[stash 中有哪些内容] **今日开始建议**：</p>

<p>- [基于当前状态给出最优先的开始点] - [提示可能需要处理的遗留问题] 保持简洁，5分钟内可读完。</p>

<h2>§20.4 实战案例二：/perf-audit — 性能审计</h2>

<p>背景：定期对修改过的文件进行性能分析，发现潜在的性能问题。</p>

<p>路径：.claude/skills/perf-audit/SKILL.md （项目级） --- 对代码进行性能审计，发现 N+1 查询、内存泄漏、不必要的重渲染等问题。</p>

<p>当用户说&quot;审计性能&quot;、&quot;检查性能问题&quot;、&quot;这里有性能问题吗&quot;时触发。</p>

<p>argument-hint: [文件或目录路径] --- 对 $ARGUMENTS（若无参数则分析最近修改的文件）进行性能审计。</p>

<p>## 审计维度 ### 数据库/API 层 - N+1 查询（循环中的查询调用） - 缺失索引指标（大表全扫描） - 未使用的 JOIN 或过度 SELECT ### 前端层（如适用） - 不必要的重渲染（缺失 memo/useMemo/useCallback） - 大列表未虚拟化 - 阻塞主线程的同步操作 ### 通用 - 内存泄漏风险（未清理的监听器、定时器） - 冗余计算（可缓存的结果被重复计算） ## 输出格式 ### 性能审计报告 **审计文件**：[文件列表] **高风险问题**（需立即修复）：</p>

<p>- [ ] 问题描述 → 具体位置 → 修复建议 **中风险问题**（建议改进）：</p>

<p>- [ ] 问题描述 → 具体位置 → 修复建议 **低风险优化**（可选）：</p>

<p>- [ ] ... **总体评分**：[1-10，并简要说明]</p>

<h2>§20.5 实战案例三：/onboard — 项目上手引导</h2>

<p>背景：新成员加入项目或自己切换到陌生代码库时，快速获得项目全景认知。</p>

<p>路径：.claude/skills/onboard/SKILL.md （项目级，提交到 git） --- 为新加入项目的开发者提供代码库全景解读。</p>

<p>当用户说&quot;帮我了解这个项目&quot;、&quot;项目结构是什么&quot;、&quot;怎么开始贡献&quot;时触发。</p>

<p>--- 对当前代码库进行全景分析，为新成员生成上手指南。</p>

<p>## 分析步骤 1. 读取 README.md、CONTRIBUTING.md、CLAUDE.md（如有） 2. 分析目录结构（不超过两层深度） 3. 识别核心入口文件（main、index、app 等） 4. 查看最近 20 条提交了解当前开发重点：`git log --oneline -20` 5. 识别主要依赖和技术栈（package.json 或 requirements.txt） ## 输出格式 # 项目上手指南 ## 项目概述 [一段话描述项目是什么、解决什么问题] ## 技术栈 [列出主要语言、框架、数据库] ## 目录结构 [关键目录说明，重点标注&quot;最应先读的文件&quot;] ## 本地运行 [从 README 提取的启动步骤] ## 当前开发重点 [基于最近提交分析的正在进行的工作] ## 新手建议 [从哪个文件开始读、哪个功能模块最适合入手贡献]</p>

<h2>§20.6 上下文预算管理</h2>

<p>Skills 的 description 始终在上下文中，但总预算有限：</p>

<p>指标 | 值</p>

<p>动态预算 | 上下文窗口的 2%</p>

<p>兜底值 | 16,000 字符</p>

<p>超出时 | 较低优先级的 Skill 可能被排除</p>

<p>扩大预算设置环境变量 SLASH_COMMAND_TOOL_CHAR_BUDGET=&lt;字符数&gt; 核心建议 Skills 太多导致部分 Skill 不触发时，运行 /context 检查是否有 Skill 被排除的警告。可以通过将不常用的 Skill 设置 disable-model-invocation: true 来节省 description 预算，同时保留手动调用能力。</p>

<h2>§20.7 技能库的长期维护策略</h2>

<p>个人技能库是需要持续维护的资产，建议遵循以下原则：</p>

<p>定期回顾：每季度审视一次，删除不再使用的 Skill，优化触发不准的 description。</p>

<pre><code>版本控制：用户级技能库（~/.claude/skills/ ）建议单独建一个 git 仓库管理，在多台机器间同步。项目级</code></pre>

<p>技能库随项目版本控制，记录在 CHANGELOG 中。</p>

<p>技能复用：将通用步骤提取到用户级 Skill，项目级 Skill 只保留项目特定的配置（如具体的部署命令、特定的代 码风格要求）。</p>

<p>团队共建：项目级 Skills 应像代码一样经过 Code Review，确保命令的安全性和预期行为符合描述（遵循最小 意外原则）。</p>

<p>一个务实建议：不要一开始就设计&quot;完美的技能库&quot;。从解决眼前的重复问题开始，用最简单的 SKILL.md 写下 你第三次重复的提示词，积少成多。三个月后回头看，你会拥有一套真正贴合自己工作方式的专属工具集。</p>
