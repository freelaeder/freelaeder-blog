## 笔记整理

<p>Claude Code 拥有读写文件系统、执行命令的能力，配置不当可能带来严重的安全风险。本节介绍从 权限规则到危险操作防护的完整安全配置体系，帮助你在效率与安全之间找到正确的平衡点。</p>

<p>默认安全模型 Claude Code 从只读权限开始，每次需要修改文件或运行命令时都会请求显式许可。理解两个关键约束：</p>

<p>写入范围限制：Claude Code 只能写入启动目录及其子目录。它可以读取父目录中的文件（如系统库、npm 依 赖），但无法修改。这是一个硬性边界，不通过配置绕开。</p>

<p>默认拦截的危险命令：`curl`、`wget`（网络请求）、`dd`（磁盘操作）等命令默认需要手动确认，防止未经审 核的外部数据进入系统。</p>

<p>权限规则体系 权限系统支持三级规则，求值顺序固定为 deny &gt; ask &gt; allow，第一个匹配规则立即生效：</p>

<p>规则 | 防护对象 | 风险说明</p>

<p>Read(./.env) 环境变量文件 API Key、数据库密码 Read(./.env.*) 各环境配置 .env.production、.env.local Read(./secrets/**) 密钥目录私钥、证书 Read(~/.ssh/id_rsa) SSH 私钥服务器访问凭证 Read(~/.aws/credentials) AWS 凭证云资源访问权限 Read(~/.kube/config) K8s 配置集群控制权限 WebFetch 所有外部请求防止数据外传 注意 deny 规则不是可选优化，而是安全基线。一旦 Claude 读取了包含 API Key 的 .env 文件，该 Key 就会出现在 对话上下文中，存在通过日志或对话历史泄露的风险。</p>

<p>沙盒化：最强安全边界 权限规则是声明式的，有被绕过的可能。沙盒（`/sandbox`）是 OS 级隔离，提供更强的安全保证：</p>

<p>{ }, } } } 沙盒的实际效果：根据 Anthropic 内部使用数据，启用沙盒后权限提示数量减少约 84%——因为沙盒内部的操 作都在预定义的安全范围内，不需要额外确认。这意味着开启沙盒不是限制效率，而是通过预定义边界提升效 率。</p>

<p>这个标志跳过所有权限提示，是 Claude Code 最危险的运行模式之一。它的存在有合理的使用场景，但必须严 格限定。</p>

<p># 危险：直接 pipe 不受信任的内容 # 安全：先下载到本地，人工审查后再分析 # 人工检查文件内容后再执行：</p>

<p>Claude Code 内置了多项防护：上下文感知分析、命令注入消毒、默认拦截 `curl`/`wget` 等。但用户层面的 习惯同样重要：</p>

<p>执行前人工审查每条建议的命令，特别是涉及网络或文件删除的操作 不直接 pipe 不受信任内容给 Claude 与外部 Web 服务交互时优先在虚拟机或容器中操作 遇到可疑行为立即使用 `/feedback` 报告 通过 Hook 审计配置变更 使用 ConfigChange Hook 阻止 Claude 在会话期间修改安全配置：</p>

<p>对于需要强制执行的组织标准，使用 Managed Settings（托管配置）：</p>

<p>平台 | 托管配置路径</p>

<p>macOS /Library/Application Support/ClaudeCode/managed-settings.json Linux / WSL /etc/claude-code/managed-settings.json 托管配置中可以设置 `&quot;allowManagedPermissionRulesOnly&quot;: true`，阻止用户通过本地配置覆盖组织级安全规 则。</p>

<p>核心建议 安全配置的最佳实践顺序：先配置 deny 规则（敏感文件和危险命令），再配置 ask 规则（有风险但合法的操 作），最后配置 allow 规则（已知安全的常用命令）。从最严格开始，逐步放开，而不是从开放开始，事后收 紧。</p>
