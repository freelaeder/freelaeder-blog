## 笔记整理

<p>从零打造属于自己的斜杠命令库，是将 Claude Code 从通用 AI 助手升级为专属工作流引擎的关键一 步。本节覆盖目录结构、新旧两种创建方式，以及六个可直接复用的完整命令模板——涵盖 Git 提交、 Code Review、每日站会等最常用场景。</p>

<h2>§18.1 两种创建方式：旧路径与新 Skills 格式</h2>

<p>Claude Code 支持两种创建自定义命令的方式，它们功能等价，可以共存：</p>

<p>不推荐 推荐 新格式：Skills（推荐） 旧格式：Commands（继续有效） 支持：Frontmatter 控制字段、辅助资源目录、子 支持：纯 Markdown 指令、$ARGUMENTS 参数替换 Agent 触发、自动激活 更简单，但功能有限 同名时优先于 Commands 核心建议 旧有 .claude/commands/ 文件无需迁移，继续有效。新建命令建议使用 Skills 格式以获得完整功能支持。</p>

<h2>§18.3 六个实用命令完整模板</h2>

<p>模板一：/commit — 规范化 Git 提交</p>

<pre><code>路径：~/.claude/skills/commit/SKILL.md （用户级，全项目可用）</code></pre>

<p>--- description: 创建规范化的 Git 提交，遵循 Conventional Commits 格式 --- 分析当前暂存区的变更，创建一个规范化提交。</p>

<p>## 提交格式 遵循 Conventional Commits 规范：</p>

<p>## 步骤 1. 运行 `git diff --cached` 查看暂存变更 2. 分析变更的性质和影响范围 3. 如果用户提供了 $ARGUMENTS，将其作为提交描述的参考 4. 生成提交信息草稿，展示给用户确认 5. 用户确认后执行 `git commit -m &quot;...&quot;` ## 注意事项 - 不要跳过 pre-commit hooks，不要使用 --no-verify - 描述用中文或英文（跟随项目惯例） - body 部分说明&quot;为什么&quot;而非&quot;做了什么&quot; 调用：/commit 或 /commit 修复登录页面的空指针异常 模板二：/review-pr — Code Review 路径：.claude/skills/review-pr/SKILL.md （项目级，建议提交到 git） --- description: 对 Pull Request 进行代码审查，检查代码质量、安全性和最佳实践 --- ## PR 上下文 ## 审查任务 对以上 Pull Request 进行全面代码审查，关注：</p>

<p>### 必检项 1. **功能正确性**：逻辑是否正确，边界情况是否处理 2. **安全性**：注入、权限绕过、敏感信息泄露风险 3. **性能**：N+1 查询、不必要的循环 4. **可读性**：命名是否清晰，复杂逻辑是否有注释 ### 输出格式 ### 总体评估 [一句话概括] ### 必须修复 - [ ] ... ### 建议改进 - [ ] ... ### 优点 - ... ### 结论 模板三：/daily-standup — 每日站会汇报 --- description: 生成每日站会汇报，汇总昨天的工作和今天的计划 --- ## 获取工作记录 收集以下信息：</p>

<p>- 昨天的 git 提交：!`git log --since=&quot;yesterday 00:00&quot; --until=&quot;today 00:00&quot; --oneline --author - 今天的 git 提交：!`git log --since=&quot;today 00:00&quot; --oneline --author=&quot;$(git config user.name)&quot; - 当前分支状态：!`git status --short` ## 生成汇报 根据收集到的信息生成站会汇报：</p>

<p>**昨天完成**：</p>

<p>- [列举提交涉及的功能/修复] **今天计划**：</p>

<p>- [根据当前 WIP 和 TODO 推断] **阻碍**：</p>

<p>- 无（如果 git 状态显示有冲突或问题则列出） 保持简洁，每项不超过一行。</p>

<p>调用：/daily-standup，每天上班第一件事。</p>

<p>模板四：/test — 智能测试运行 --- description: 运行测试并分析失败原因，提供修复建议 argument-hint: [测试文件或测试名称] --- 运行测试 $ARGUMENTS（若无参数则运行全部测试）。</p>

<p>## 执行步骤 1. 检测项目测试框架（package.json 中的 test 脚本，或 pytest.ini） 2. 运行测试：</p>

<p>- 若有参数：运行指定测试 - 若无参数：运行全套测试 3. 分析结果：</p>

<p>- 通过：报告测试数量和耗时 - 失败：分析每个失败原因 - 是代码 Bug？找到出错位置并提供修复建议 - 是测试本身的问题？指出并建议更新 - 是环境问题？说明如何修复环境 ## 输出格式 测试结果：X 通过 / Y 失败 / Z 跳过 耗时：Xs 失败分析：[每个失败的详细分析] 建议：[具体的修复建议] 模板五：/deploy — 生产部署 路径：.claude/skills/deploy/SKILL.md （项目级，含完整安全检查） --- description: 将应用部署到生产环境 --- 部署 $ARGUMENTS 环境。</p>

<p>## 部署前检查 1. 确认当前分支是 main 或 release/* 2. 确认本地无未提交变更：`git status` 3. 确认所有测试通过：`npm test` 或 `pytest` 4. 确认 CHANGELOG 已更新 ## 部署步骤 5. 构建应用：`npm run build` 6. 运行集成测试：`npm run test:e2e` 7. 推送到部署目标（根据 $ARGUMENTS 选择）：</p>

<p>8. 验证部署：检查健康检查端点 9. 通知团队（如有 Slack webhook） ## 失败处理 任何步骤失败立即停止，报告失败原因，不要继续部署。</p>

<p>模板六：/fix-issue — 从 Issue 直接开发 --- description: 读取 GitHub Issue 并实现修复，包含测试和提交 --- ## 步骤 2. 理解需求，识别需要修改的代码区域 3. 实现修复，保持代码风格与现有代码一致 4. 编写或更新测试覆盖新逻辑 5. 运行测试确认通过 6. 创建 Conventional Commits 格式的提交 7. 推送并创建 PR，关联 Issue：</p>

<p>## 注意 - 优先修改最小必要范围，不引入无关变更 - PR 描述中必须包含 &quot;Closes #issue-number&quot; 自动关闭 Issue 调用：/fix-issue 42 核心建议 这六个模板覆盖了开发者 80% 的日常高频场景。建议将 /commit、/daily-standup、/explain-code、/fix- issue 放在用户级（~/.claude/skills/ ），将 /review-pr、/test、/deploy 放在项目级 （.claude/skills/ ）并提交到 git 供团队共享。</p>
