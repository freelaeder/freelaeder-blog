## 笔记整理

<p>Claude Code 不是聊天机器人，是自主代理。写给它的 Prompt 更接近&quot;任务规格说明书&quot;而非&quot;对话消 息&quot;。掌握四要素框架，你的每一条指令都能产出精准、可验证的结果。</p>

<p>核心认知：上下文窗口是稀缺资源 在理解 Prompt 技巧之前，必须先理解 Claude Code 的核心约束：上下文窗口会快速填满，且上下文越满，性 能越差。这意味着一切 Prompt 策略都服从于一个目标——在最少的 token 消耗下，获得最精准的输出。</p>

<p>这也是为什么&quot;写详细的 Prompt&quot;和&quot;写冗长的 Prompt&quot;是两回事。好的 Prompt 是精确的，每个词都在压缩信 息密度；坏的 Prompt 是啰嗦的，在用自然语言描述 Claude 自己读代码就能知道的事情。</p>

<p>四要素框架 高质量 Prompt 需要覆盖以下四个维度——并非每条 Prompt 都需要全部四个，但越复杂的任务，越需要把四个 要素都想清楚：</p>

<p>范围 验证标准 指向 参照 Scope → Direction → Reference → 要素一：范围（Scope） 作用：告诉 Claude 任务的边界，防止它&quot;发散&quot;到不相关的地方。</p>

<p>维度 | 差的写法 | 好的写法</p>

<p>文件范围 &quot;给这个项目加测试&quot; &quot;给 src/utils/validator.ts 加测试&quot;</p>

<p>功能范围 | &quot;优化性能&quot; | &quot;优化 getUsersWithOrders() 的查询性能&quot;</p>

<p>约束范围 | &quot;加一个功能&quot; | &quot;不引入新的 npm 依赖，使用现有的 axios&quot;</p>

<p>进阶技巧：让 Claude 来采访你 对于大型功能，与其自己绞尽脑汁写完整的需求文档，不如先让 Claude 采访你——它比你更清楚一个功能实现 需要考虑哪些边界情况：</p>

<p>我想构建一个多租户权限系统。请用 AskUserQuestion 工具详细采访我。</p>

<p>采访内容应包括：</p>

<p>- 技术实现方案（数据库 schema、缓存策略） - 权限粒度（资源级别 vs 操作级别） - 边界情况（超级管理员、权限继承、冲突处理） - 性能顾虑（权限检查的频率和成本） 不要问显而易见的问题，要深入挖掘我可能没有想到的难点。</p>

<p>持续采访直到覆盖所有方面，然后将完整的规格说明写入 SPEC.md。</p>

<p>采访结束后，开一个全新的 Session 来执行规格说明。新 Session 有干净的上下文，专注于实现本身，而不是 被采访过程中的大量对话&quot;污染&quot;。</p>

<p>CLAUDE.md：让好 Prompt 只写一次 如果某些约束条件是项目级别的（比如&quot;总是用 ES modules&quot;、&quot;commit message 用中文&quot;），把它们写进 CLAUDE.md，而不是每次 Prompt 都重复一遍。CLAUDE.md 是每次会话开始时 Claude 自动读取的配置文 件。</p>

<p># 代码风格 - 使用 ES modules（import/export），不用 CommonJS（require） - 尽可能解构导入：import { foo } from &#x27;bar&#x27; - TypeScript 严格模式，所有公共 API 必须有类型注解 # 测试规范 - 测试文件放在与源文件同目录，命名为 *.test.ts - 优先运行单个测试：npx jest --testPathPattern=xxx - 不使用 mock，优先集成测试 # Git 规范 - Commit message 遵循 Conventional Commits 规范（中文） - PR 描述必须包含改动摘要和测试计划 核心建议 CLAUDE.md 的黄金法则：对每一行问自己&quot;删掉这行会让 Claude 犯错吗？&quot;如果不会，就删掉。太长的 CLAUDE.md 会导致 Claude 忽略真正重要的指令。</p>
