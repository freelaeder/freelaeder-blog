## 笔记整理

<p>系统性调试不是运气，而是方法。本节围绕&quot;根因优先&quot;这一核心原则，介绍从日志缩小范围到错误模 式归档的完整调试工作流，让 Claude Code 成为你最有力的调试搭档。</p>

<p>根因优先原则 调试最大的陷阱不是找不到问题，而是找到了症状就急于修复。Claude Code 的默认倾向是尝试修复它看到的 第一个错误，这往往导致治标不治本——症状消失了，根因依然潜伏。</p>

<p>调试黄金法则：在定位到根本原因之前，拒绝执行任何修复。压制错误（try-catch 吞掉异常、条件分支绕过问 题）不是修复，是技术债务。</p>

<p>系统性调试遵循四步法：</p>

<p>1 | 稳定复现</p>

<p>在修复之前，先确认能在受控条件下稳定触发问题。无法复现的 bug 无法验证修复效果。将复现步骤写 成测试用例，作为修复的验证基准。</p>

<p>2 | 隔离根因</p>

<p>通过日志、断点或二分法缩小范围，找到真正的错误源头，而不只是症状出现的位置。错误堆栈往往指向 症状，不一定指向根因。</p>

<p>3 | 针对性修复</p>

<p>仅修改导致根因的代码，不修改无关的&quot;顺手改&quot;。范围越小，引入新问题的风险越低。</p>

<p>4 | 验证并回归</p>

<p>运行原有复现用例确认修复有效，同时运行完整测试套件确认没有引入新问题。</p>

<p>高效 Bug 定位 Prompt 模板 给 Claude 的 bug 报告质量直接决定调试效率。以下是三种典型场景的 Prompt 模板：</p>

<p>基础 Bug 报告（含完整上下文）：</p>

<p>运行 npm test 时出现以下错误：</p>

<p>[粘贴完整错误信息和堆栈追踪] 复现步骤：</p>

<p>1. 登录后等待 30 分钟 2. 尝试访问 /dashboard 页面 修复它并验证构建成功。处理根本原因，不要压制错误。</p>

<p>精确定位 Bug（先写失败测试）：</p>

<p>用户反映 session 超时后登录失败。</p>

<p>检查 src/auth/ 中的认证流程，特别是 token 刷新逻辑。</p>

<p>先写一个能复现问题的失败测试，然后修复它，最后确认测试通过。</p>

<p>性能问题（N+1 查询）：</p>

<p>这个查询在数据量超过 1000 条时响应时间超过 5 秒。</p>

<p>分析 getUsersWithOrders() 函数的时间复杂度， 识别 N+1 查询或其他性能瓶颈， 提出优化方案并估算预期改善幅度。</p>

<p>逐步日志缩小范围法 对于路径长、状态复杂的 bug，最有效的策略是分轮添加日志，逐步将候选范围缩小到个位数的代码行。</p>

<p>不推荐 推荐 一次性在所有地方加日志，输出几百行，难以找到关键 分轮添加日志：第一轮覆盖主路径，分析输出后，第二 信息，浪费大量 token。</p>

<p>轮聚焦到可疑区域，直到定位到具体行。</p>

<p>第一轮：覆盖主调用路径 在 processPayment() 函数的关键路径上加上全面的日志， 记录每个步骤的输入输出和中间状态。</p>

<p>第二轮：根据日志聚焦 [粘贴第一轮日志输出] 根据这些日志，在 validateCard() 和 chargeCard() 之间继续缩小范围， 重点记录 amount 字段在每次转换时的值。</p>

<p>自动化日志分析（管道模式）：</p>

<p>npm run test 2&gt;&amp;1 | tee outfile | claude -p &quot;分析这个测试输出，找出失败的根本原因&quot; 这种方式让测试输出实时传给 Claude 分析，不需要手动复制粘贴大量日志。</p>

<p>使用 Plan Mode 安全调试 对于涉及核心模块的复杂 bug，先在 Plan Mode 分析，防止 Claude 急于修改而破坏正常工作的代码：</p>

<p>粘贴错误信息和堆栈追踪，彻底调查这个 bug， 不要做任何修改，只分析根本原因并制定修复计划。</p>

<p>确认分析结论和修复计划之后，切换回 Normal Mode 执行。这个习惯能避免&quot;修复一个 bug，引入三个 bug&quot;的 恶性循环。</p>

<p>Bug 模式归档到 CLAUDE.md 每次解决一个非显而易见的 bug，都应该将其模式写入 CLAUDE.md 的&quot;已知 Bug 模式&quot;章节。这样下次遇到类 似症状，Claude 能直接跳过排查过程，快速定位到根因。</p>

<p># 已知 Bug 模式 ## Session 超时问题 - 症状：用户登录后 30 分钟报 401 错误 - 根因：Redis session 的过期时间设置没有滑动窗口 - 解决：在 src/auth/session.ts 中启用 rolling session ## 支付金额精度问题 - 症状：小数计算出现浮点误差（如 $9.99 变成 $9.990000000001） - 根因：使用 JavaScript 原生 float 做货币计算 - 解决：在 src/payment/ 中统一使用 Decimal.js 核心建议 每个归档的 bug 模式都应包含三要素：症状（外部可观察到的现象）、根因（真正的代码问题）、解决方案（具 体的文件路径和修改方向）。缺少任何一项都会降低归档的价值。</p>

<p>UI Bug 的视觉调试 对于布局、样式相关的 bug，截图胜过千言万语。直接粘贴截图给 Claude，比文字描述效率高出数倍：</p>

<p>[粘贴出现 bug 的页面截图] 这个布局问题出在哪里？</p>

<p>分析 CSS 根本原因并修复。</p>
