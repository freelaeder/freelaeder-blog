## 笔记整理

<p>上下文窗口是 Claude Code 最关键的资源。它不是无限的，也不是越满越好——上下文越满，性能越 差，Claude 越容易&quot;忘记&quot;早期的重要指令，犯更多低级错误。本节聚焦三个实战维度：理解 token 消 耗的来源、掌握 /compact /clear /rewind 三大上下文管理命令的使用时机，以及用子代理隔离上下文 的高级策略。</p>

<p>什么在消耗你的上下文 每个会话的上下文窗口包含多种内容，按占用量从高到低排列：</p>

<p>内容类型 | 典型占用 | 控制方式</p>

<p>对话历史（所有消息、工具调用结果） 会话期间持续增长 /compact 或 /clear</p>

<p>读取的文件内容 | 每次 Read 追加 | 按需读取，不要预防性读取</p>

<p>命令输出（Bash 结果） | 取决于输出量 | 过滤输出（Hook），限制行数</p>

<p>CLAUDE.md 文件 | 每次会话固定加载 | 保持简洁（&lt;200 行）</p>

<p>Auto Memory（MEMORY.md） 前 200 行自动加载精简记忆文件</p>

<p>MCP 工具定义 | 每个 MCP 服务器贡献 | 禁用不用的 MCP 服务器</p>

<p>Skills 描述 | 按需加载 | 仅加载当前需要的 Skill</p>

<p>查看上下文使用情况：</p>

<p>/context # 查看当前上下文的各组成部分及其占用 /cost # 查看当前 session 的 token 使用和预估费用 /stats # 查看使用模式统计（Max/Pro 订阅者） /compact：智能压缩，保留精华 /compact 是最温和的上下文管理命令。它让 Claude 对当前对话历史进行智能摘要：保留关键信息（决策、修 改的文件、未解决的问题），清理冗余（重复的工具输出、探索过程中的中间步骤）。</p>

<p>使用时机：</p>

<p>完成一个子任务，准备开始下一个子任务（例如：完成了 API 接口，现在要写测试） 上下文使用率超过 50%，但任务还没完成 对话中有大量探索性的来回，但核心结论已经明确 带焦点的定向压缩：</p>

<p># 普通压缩：Claude 自行判断保留什么 # 定向压缩：告诉 Claude 什么最重要 /compact 重点保留 API 接口的改动和测试命令 在 CLAUDE.md 中自定义压缩行为：</p>

<p>当使用 /compact 压缩时，始终保留：</p>

<p>- 已修改文件的完整列表 - 当前任务的关键测试命令 - 已确认的架构决策 - 正在处理的 Bug 描述和当前进展 核心建议 /compact 适合&quot;任务连续，但历史变冗余&quot;的场景。压缩后，Claude 对当前任务状态的认知保持完整，但上下 文占用大幅减少，可以继续在当前会话中工作。</p>

<p>/clear：彻底清空，全新开始 /clear 会清空整个对话历史，但保留 CLAUDE.md 和 Auto Memory——它们会在下次消息时重新加载。效果 等同于开一个全新的会话。</p>

<p>使用时机：</p>

<p>切换到完全不相关的新任务（从修 Bug 切换到写文档） 多次纠正后 Claude 陷入&quot;死循环&quot;（连续纠正 3 次以上） 会话中积累了大量失败尝试的历史，严重影响后续质量 当前任务完成，准备开始下一个独立任务 不推荐 推荐 正确使用 /clear 的节奏：</p>

<p>错误做法：</p>

<p>纠正 1 次 → 继续当前会话 在一个会话中连续处理：</p>

<p>纠正 2 次 → 考虑 /compact 压缩重来 认证模块重构 → 支付 Bug → 文档更新 纠正 3 次 → 必须 /clear，重新思考 Prompt → 上下文充满无关信息，质量持续下滑 注意 在 /clear 之前，先用 /rename 给当前会话命名，以便日后用 /resume 恢复。清空了对话历史但 CLAUDE.md 里记录了关键决策的场景，是最理想的 /clear 使用方式。</p>

<p>/rewind：时间回溯，选择性撤销 /rewind （也可以按两次 Esc ）让你回到会话中的某个历史检查点，支持两种操作：</p>

<p>从此处继续：还原到该检查点，丢弃之后的所有操作（适合&quot;做错了方向，从这里重来&quot;） 从此处摘要：将该检查点之后的消息压缩成摘要，前面的完整上下文保留（适合局部压缩） 使用时机：</p>

<p>Claude 刚做了一系列你不满意的修改，想直接撤销（而不是让它&quot;改回去&quot;） 想从某个分支点重试不同的方案 测试不同的 Prompt 策略，不想污染原有对话历史 会话检查点的工作原理：</p>

<p>Claude Code 在每次文件编辑前自动创建检查点。这意味着每次 Edit/Write 操作都有一个可以回退的状态。</p>

<p># 配合会话分支使用 # 先保存主线，再开分支尝试不同方案 三个命令的对比选择</p>

<p>命令 | 保留内容 | 适用场景 | 类比</p>

<p>/compact 关键信息摘要 + CLAUDE.md 任务连续，历史变冗余整理书桌，留下重要文件 /clear 仅 CLAUDE.md + Memory 切换任务，全新开始清空书桌，只留手册 /rewind 回到某个历史状态方向错误，回退重试撤销操作，时间倒流 Token 消耗意识：五个实操习惯 良好的 token 消耗意识，能在不影响质量的前提下大幅降低使用成本：</p>

<p>1 | 按需读取，不预防性读取</p>

<p>不要一开始就让 Claude 读整个 src/ 目录。用 Glob + Grep 先缩小范围，只 Read 真正需要的 3-5 个文 件。</p>

<p>2 | 过滤测试输出</p>

<p>测试输出可能有几千行，但失败信息只有几十行。通过 Hook 或管道过滤，只把失败信息传给 Claude：</p>

<p>3 | 大文件分段读取</p>

<p>对于超过 500 行的文件，用 offset 和 limit 参数只读关键段落，而不是整个文件。</p>

<p>4 | 保持 CLAUDE.md 简洁</p>

<p>CLAUDE.md 每次会话都全量加载。超过 200 行不仅浪费 token，还会导致 Claude 忽略其中的关键规 则。目标：30-100 行精华内容。</p>

<p>5 | 按任务选择模型</p>

<p>日常编码（约 80% 任务）使用 Sonnet，性价比最高；复杂架构决策使用 Opus；简单子代理任务使用 Haiku（成本极低）。通过 /model 命令随时切换。</p>

<p>子代理隔离上下文：最高级的上下文管理 子代理（Subagent）在独立的上下文窗口中运行，完成研究后向主会话汇报摘要。这使得&quot;探索性调查&quot;工作不 会污染主会话的上下文：</p>

<p># 让子代理调查，不污染主上下文 &quot;使用子代理调查我们的认证系统如何处理 token 刷新， 以及是否有现有的 OAuth 工具可以复用。</p>

<p>给我一个简洁的调查报告，不需要详细的探索过程。&quot; # 子代理另一个典型用途：并行调查 &quot;使用子代理检查 src/payment/ 是否存在边界情况和安全问题&quot; &quot;使用另一个子代理分析 src/auth/ 的测试覆盖率&quot; 子代理的核心优势：</p>

<p>探索工作不消耗主会话 token 多个子代理可以并行工作，加速大型调查 子代理失败不影响主任务进行 子代理可以指定轻量模型（Haiku），降低成本 核心建议 最佳实践：将&quot;调查理解&quot;工作交给子代理，将&quot;决策执行&quot;工作留在主会话。主会话应该保持干净的上下文，只 包含做决策和执行修改所需的信息。</p>

<p>会话管理：命名、恢复、分支 良好的会话管理习惯，让你在多个任务之间自如切换，不丢失任何工作：</p>

<p># 启动时命名会话 # 会话中随时重命名 # 恢复已有会话 claude --continue # 恢复最近的会话 claude --resume # 交互式选择要恢复的会话 claude --resume auth-refactor # 按名字直接恢复 claude --from-pr 123 # 恢复与某 PR 关联的会话 # 创建会话分支（尝试不同方案，不影响主线） 跨任务工作的最佳实践：将每个独立任务放在命名会话中。任务之间用 /clear 清空上下文。需要回到某个任 务时，用 claude --resume &lt;名称&gt; 恢复，恢复后会自动重新加载 CLAUDE.md，干净整洁。</p>

<p>上下文管理的整体策略 把上下文管理融入日常工作节奏：</p>

<p>时机 | 动作 | 原因</p>

<p>任务开始 | 确认 CLAUDE.md 已就位 | 持久规则无需反复口头说明</p>

<p>完成子任务 /compact 清理探索历史，保留核心状态 切换新任务 /rename + /clear 先保留旧会话，再全新开始</p>

<p>Claude 方向跑偏 | /rewind 回退检查点 | 比让它&quot;改回去&quot;更干净</p>

<p>连续纠正 3 次 /clear + 重写 Prompt 重新开始，而非在错误基础上叠加</p>

<p>需要调查分析 | 使用子代理 | 隔离探索上下文，不污染主会话</p>
