## 笔记整理

<p>探索→规划→实现→提交，这四个阶段构成了 Claude Code 最高效的工作节奏。每个阶段有专属的工 具、模式和 Prompt 策略，打通这条流水线，你的开发效率将产生质的飞跃。</p>

<p>为什么需要&quot;四阶段&quot;？</p>

<p>很多开发者第一次使用 Claude Code 时，习惯直接把问题甩进去：&quot;帮我实现 Google OAuth 登录&quot;。Claude 会 立刻开始写代码，但问题往往随之而来：它不了解你的 session 管理方式，不知道你用了哪个认证库，也不清 楚哪些文件需要修改。结果是产出了一堆孤立的代码片段，需要你手动拼装，和不用 AI 相比并没有节省多少时 间。</p>

<p>四阶段工作法的核心逻辑是：把认知负担前置，把执行动作后置。先让 Claude 充分理解项目现状（探索），再 制定清晰的行动计划（规划），然后全速执行（实现），最后固化成果（提交）。每个阶段的工具和权限模式都有 明确的边界，防止 Claude 在信息不足时&quot;盲目行动&quot;。</p>

<p>核心认知：Plan Mode（Shift+Tab 循环切换）是四阶段工作法的基础设施。在探索和规划阶段保持 Plan Mode，Claude 只读不写；切换到 Normal 或 Auto-Accept Mode 后才开始实际修改。</p>

<p>阶段一：探索（Explore） 探索阶段的目标是让 Claude 对代码库建立足够的上下文，以便后续规划准确。这个阶段在 Plan Mode 下运行 ——Claude 只能使用只读工具（Read、Grep、Glob），不会修改任何文件。</p>

<p>读取 src/auth 目录，了解我们的 session 处理和登录机制。</p>

<p>同时查看 .env.example 和配置文件，理解 secrets 的管理方式。</p>

<p>3 | 追问关键细节</p>

<p>探索不是一次性的。根据 Claude 的初步反馈，继续追问感兴趣的部分：</p>

<p>session 是存在 Redis 还是数据库？token 刷新逻辑在哪个文件里？</p>

<p>现有的认证中间件支持第三方 OAuth 吗？</p>

<p>4 | 确认探索充分</p>

<p>当你觉得 Claude 已经理解了足够的背景，转入规划阶段。判断标准：Claude 能准确描述&quot;改动会影响哪 些文件&quot;。</p>

<p>核心建议 探索阶段不要急，这里每多花 2 分钟，后续的实现阶段就能少返工 20 分钟。大型功能（涉及 5 个以上文件） 至少探索 2-3 轮对话再进入规划。</p>

<p>阶段二：规划（Plan） 规划阶段依然在 Plan Mode 下进行。Claude 基于探索结果制定具体的实施方案——哪些文件需要修改、修改什 么、执行顺序是什么。这个计划你可以直接编辑。</p>

<p>1 | 提出需求并请求计划</p>

<p>我想加入 Google OAuth 登录。</p>

<p>基于你刚才对认证模块的了解，哪些文件需要改动？</p>

<p>请给我一个分步骤的实施计划，包括每步修改的文件名和具体内容。</p>

<p>2 | 在编辑器中修改计划</p>

<p>按 Ctrl+G 可以在默认文本编辑器中打开计划内容，直接编辑、删除步骤或添加约束条件，保存后返回</p>

<p>3 | 反复迭代直到满意</p>

<p>第三步中你提到要修改 middleware/auth.ts， 但我们的 OAuth callback 处理应该放在 routes/ 里， 请调整计划。</p>

<p>4 | 确认计划，切换执行模式</p>

<p>计划确认后，按 Shift+Tab 切换到 Auto-Accept Mode（⏵⏵ accept edits on）。后续 Claude 会自动应 用文件修改，无需逐一确认。</p>

<p>Pro Tip：接受计划后，Claude Code 会根据计划内容自动为本次会话命名（如&quot;feat: google-oauth- integration&quot;），方便后续通过 /sessions 恢复。</p>

<p>阶段三：实现（Implement） 实现阶段是真正的执行环节。Claude 按照既定计划逐步完成文件修改、测试编写和验证。你的角色从&quot;指挥 者&quot;变成&quot;监督者&quot;。</p>

<p>1 | 发出执行指令（含验证标准）</p>

<p>按照你的计划实现 Google OAuth 登录流程。</p>

<p>为 OAuth callback handler 写单元测试， 运行 npm test，确保所有测试通过。</p>

<p>不要引入新的 npm 依赖，使用现有的 passport.js 配置。</p>

<p>2 | 监控执行过程</p>

<p>在 Auto-Accept Mode 下，Claude 自动应用文件修改，但 Shell 命令仍需确认。观察每次修改是否符合 计划，发现偏差时按 Esc 打断并纠正。</p>

<p>测试失败了，错误信息如下：</p>

<p>[粘贴错误] 先理解根本原因，再修复。不要压制错误。</p>

<p>4 | 需要回滚时使用 Checkpoint</p>

<p>Claude Code 在每次文件修改前自动创建检查点。按两次 Esc 或运行 /rewind，可选择回滚到任意历 史状态。</p>

<p>不推荐 推荐 实现阶段给出验证标准：</p>

<p>实现阶段没有验证标准：</p>

<p>&quot;运行 npm test，确保所有测试通过，类型检查无报 &quot;实现 OAuth 登录&quot; 错&quot; → Claude 可能产出&quot;看起来对&quot;但实际无法运行的代码 → Claude 会主动验证自己的输出 阶段四：提交（Commit） 提交阶段将实现成果固化到版本控制，并通过 PR 进入团队协作流程。Claude Code 原生支持 git 操作和 gh CLI。</p>

<p>1 生成 Commit Message 检查 git diff，生成一条遵循 Conventional Commits 规范的提交信息。</p>

<p>格式：feat(auth): 描述具体改动 正文说明&quot;为什么&quot;而不只是&quot;做了什么&quot;。</p>

<p>2 创建 Pull Request 提交代码，然后创建 PR。</p>

<p>PR 描述包含：</p>

<p>- 改动摘要（3 点以内） - 测试计划（列出验证步骤） - 可能存在的风险或注意事项</p>

<p>3 | 链接会话到 PR</p>

<p>PR 创建后，会话自动链接。后续可以通过 claude --from-pr 123 恢复本次会话的完整上下文，继续处 理 Review 意见。</p>

<p>注意 提交阶段不要让 Claude 用 --no-verify 跳过 pre-commit hook，也不要用 --force 强推 main 分支。这 些操作属于高风险操作，需要你明确授权。</p>

<p>四阶段的灵活运用 四阶段不是僵化的公式。小型任务可以跳过规划阶段直接实现；紧急 Bug 修复可以在探索阶段就明确根因，省 略独立的规划对话。根据任务规模和风险级别灵活调整：</p>

<p>任务规模 | 推荐流程</p>

<p>单文件小改动（改一行日志） | 直接实现 → 提交</p>

<p>中等功能（2-5 个文件） | 简短探索 → 实现 → 提交</p>

<p>大型功能（5 个以上文件） | 充分探索 → 详细规划 → 实现 → 提交</p>

<p>架构重构 | 多轮探索 → 迭代规划（含 Ctrl+G 编辑）→ 分批实现 → 提交</p>
