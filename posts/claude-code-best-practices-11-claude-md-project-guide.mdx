## 本节重点

- 这一节围绕主题建立完整认知框架。
- 优先保留能直接迁移到日常工作中的方法。
- 把概念、流程和实践建议整理成可复用笔记。

## 笔记整理

<p>CLAUDE.md 是 Claude Code 的&quot;项目宪法&quot;——每次会话启动时自动加载，让 Claude 在新的上下文窗 口中立即了解项目规范、架构约定与工作方式。掌握四层加载机制、高效写法原则与完整模板，是告 别重复解释、实现一致性协作的关键第一步。</p>

<p>四层加载机制 CLAUDE.md 并非单一文件，而是一套按层级叠加的配置体系。Claude Code 在启动时从多个位置依次读取， 优先级由高到低如下：</p>

<p>1 | 企业托管层（Managed Policy）</p>

<p>由 IT/DevOps 管理员写入系统路径，对所有用户强制生效，且不可被排除。macOS 路径为 code/CLAUDE.md。适合全组织统一的安全与合规要求。</p>

<p>2 | 用户级层（User-global）</p>

<pre><code>存放于 ~/.claude/CLAUDE.md ，对该用户的所有项目生效。适合个人跨项目的偏好设置，如输出语言、</code></pre>

<p>提交消息格式、个人编码习惯等。</p>

<p>3 | 项目根级层（Project-root）</p>

<pre><code>存放于项目根目录的 CLAUDE.md 或 .claude/CLAUDE.md ，可提交到 git 与团队共享。这是最常用的层</code></pre>

<p>级，承载团队共识的编码规范、架构决策与工作流要求。</p>

<p>4 | 子目录延迟加载层（Sub-directory）</p>

<p>Claude 读取某个子目录中的文件时，该子目录下的 CLAUDE.md 才按需加载，而非启动时全量加载。这 种设计节省了初始上下文空间，适合在大型 monorepo 的子包中放置包级别的专项规范。</p>

<p>核心建议 Claude Code 从当前工作目录向上遍历目录树，逐层加载所有 CLAUDE.md，多层规则叠加生效。更具体的层 级不会覆盖上层，而是补充——因此层间不应有矛盾指令。</p>

<p>路径规则与模块化拆分 对于大型项目，单一 CLAUDE.md 文件难以覆盖所有场景，官方推荐将指令拆分到 .claude/rules/ 目录下的 多个文件中：</p>

<pre><code>├── CLAUDE.md # 主项目指令（&lt;200行）</code></pre>

<pre><code>└── .claude/</code></pre>

<pre><code>├── CLAUDE.md # 可选补充指令</code></pre>

<pre><code>└── rules/</code></pre>

<pre><code>├── code-style.md # 代码风格规范</code></pre>

<pre><code>├── testing.md # 测试约定</code></pre>

<pre><code>├── security.md # 安全要求</code></pre>

<pre><code>└── frontend/</code></pre>

<pre><code>└── react-patterns.md # 前端专项规范</code></pre>

<p>规则文件支持 YAML frontmatter，将指令精确绑定到特定文件路径，仅在 Claude 读取匹配路径的文件时触 发：</p>

<p>--- --- # API 开发规范 - 所有端点必须包含输入验证 - 使用标准错误响应格式：{ data, error } - 包含 OpenAPI 文档注释 此外，CLAUDE.md 支持 @path/to/file 导入语法，最多递归嵌套 5 层，适合引用 README、package.json 或外部文档：</p>

<p># 参见 @README.md 了解项目概述 高效写法原则 不推荐 推荐 &quot;正确格式化代码&quot; &quot;使用 2 空格缩进，不使用 Tab&quot; &quot;测试你的修改&quot; &quot;提交前运行 npm test，确保全部通过&quot; &quot;保持文件有序&quot; &quot;API 处理器放在 src/api/handlers/ &quot; &quot;写干净的代码&quot;（Claude 已知，冗余） 删除——让文件保持精简 六条核心原则：</p>

<p>控制行数：每个文件目标 200 行以内。更长的文件消耗更多上下文，且遵循率下降。超出时通过 @import 或 rules/ 拆分。</p>

<p>结构化分组：使用 Markdown 标题分节（项目概述、构建命令、代码规范、工作流要求），Claude 扫描结构 的方式与人类读者一致。</p>

<p>具体可验证：写可以立即执行的指令，而非抽象原则。&quot;使用 pnpm，禁止 npm 或 yarn&quot;优于&quot;选择合适的包 管理工具&quot;。</p>

<p>强调关键规则：对高优先级指令使用禁止、必须、IMPORTANT 等强调词，提高遵循率。</p>

<p>排除冗余内容：Claude 通过读代码就能推断的内容、标准语言约定、详细 API 文档（链接即可）、频繁变化 的信息——一律删除。</p>

<p>视为代码维护：纳入 git 版本控制，团队共同维护；出现问题时审查，定期修剪；通过观察 Claude 行为测 试修改效果。</p>

<p>注意 如果 Claude 持续做某件错误的事，通常说明文件太长，关键规则淹没在噪音中；如果 Claude 询问已在 CLAUDE.md 中明确回答的问题，说明措辞仍有歧义。</p>

<p>完整 CLAUDE.md 模板 以下是适用于典型全栈 Web 项目的完整模板，涵盖七个核心板块：</p>

<p># 项目概述 这是一个使用 Next.js + TypeScript + PostgreSQL 构建的 SaaS 应用。</p>

<p>主要功能：用户认证、订阅管理、内容创作工具。</p>

<p># 技术栈 - 数据库：PostgreSQL（开发用 Docker） - 包管理：pnpm（禁止使用 npm 或 yarn） # 核心命令</p>

<pre><code>pnpm dev # 启动开发服务器（端口 3000）</code></pre>

<pre><code>pnpm test # 运行单元测试</code></pre>

<pre><code>pnpm test:e2e # 运行 E2E 测试（需先启动 dev server）</code></pre>

<pre><code>pnpm build # 生产构建</code></pre>

<pre><code>pnpm db:migrate # 运行数据库迁移</code></pre>

<p># 代码规范 - 严格模式（strict: true），避免 any，使用 unknown - 优先 interface 而非 type（除非需要联合类型） - 所有 async 函数必须有错误处理 ## React 组件 - 函数式组件 + hooks，无 class 组件 - PascalCase 文件名（UserProfile.tsx） - 工具函数 camelCase（formatDate.ts） ## 目录约定 - app/ → Next.js 页面（App Router） - components/ui/ → 基础 UI 组件 - components/features/ → 功能组件 - lib/ → 工具函数和配置 - hooks/ → 自定义 hooks - types/ → 全局 TypeScript 类型 # 工作流规范 ## Git 约定 - 禁止直接推送 main，必须通过 PR + 至少 1 人 Review ## 测试要求 - 新功能必须附带单元测试 - 修复 bug 必须附带复现该 bug 的测试 - 提交前：pnpm test 全部通过 + pnpm typecheck 无错误 # 环境配置 必需的环境变量（参见 .env.example）：</p>

<p>- DATABASE_URL：PostgreSQL 连接字符串 - NEXTAUTH_SECRET：NextAuth 加密密钥（至少 32 字符） - NEXTAUTH_URL：应用 URL（开发：http://localhost:3000） - STRIPE_SECRET_KEY：Stripe API 密钥（支付功能） # 已知坑与注意事项 - E2E 测试：需先 pnpm dev，再 pnpm test:e2e - Prisma 迁移：修改 schema 后必须运行 pnpm db:migrate - Stripe Webhook 本地调试：stripe listen --forward-to localhost:3000/api/webhooks/stripe - 图片：使用 Next.js Image 组件，禁止裸 &lt;img&gt; # 禁止行为 - 禁止硬编码 API 密钥或密码 - 禁止注释掉失败的测试（修复它们） - 禁止在生产代码留 console.log - 禁止无错误处理的 fetch 调用 # 附加参考 - 详细架构说明：@docs/architecture.md - API 设计规范：@docs/api-design.md - 部署流程：@docs/deployment.md monorepo 排除技巧：在大型 monorepo 中，可通过 .claude/settings.local.json 的 claudeMdExcludes 字段排除无关子包的 CLAUDE.md，避免无关规则干扰当前工作上下文。企业托管层不受 此设置影响。</p>
