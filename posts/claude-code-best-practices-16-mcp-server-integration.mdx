## 笔记整理

<p>Model Context Protocol（MCP）是 Claude Code 连接外部服务的标准协议。通过 MCP，Claude 可 以直接操作 GitHub、Notion、数据库、内部 API 等几乎任何系统，将 AI 能力深度嵌入企业工具链。</p>

<p>MCP 架构概览 MCP 服务器是独立进程或远端服务，向 Claude Code 暴露一组可调用的&quot;工具&quot;（Tool）。Claude 通过 MCP 工 具与外部系统交互，工具命名格式为 mcp__&lt;服务器名&gt;__&lt;工具名&gt;。</p>

<p>MCP 协议层 Claude Code → JSON-RPC 2.0 → GitHub / Notion / DB… → 外部服务 API 三种传输方式 方式一：HTTP 远程服务器（推荐） 连接云端或自托管的 MCP 服务，无需在本地安装任何依赖，最简单可靠。</p>

<p>MCP + Hooks 协同的完整工作流 将 MCP 与 Hooks 结合，可以构建强大的自动化工作流。以&quot;自动化代码审查提交&quot;为例：</p>

<p>1 | Claude 使用 MCP GitHub 工具创建 PR</p>

<p>触发 PreToolUse hook，验证目标分支不是 main，PR 描述是否完整。</p>

<p>2 | PR 创建成功后，PostToolUse hook 自动触发</p>

<p>调用 CI 接口启动自动化测试，将 PR URL 发送到团队 Slack 频道。</p>

<p>3 | Claude 宣布完成时，Stop hook 介入验证</p>

<p>检查 CI 状态：若测试未完成，阻止 Claude 停止并等待；若测试失败，要求 Claude 修复。</p>

<p>4 | 所有验证通过，Notification hook 推送桌面通知</p>

<p>&quot;PR #42 已创建，CI 全绿，等待 Code Review&quot;——开发者可以放心切换到其他任务。</p>

<p>架构原则：Hooks 是&quot;护栏&quot;，MCP 是&quot;能力扩展&quot;。两者结合的正确姿势是：用 MCP 扩大 Claude 能触达的系 统范围，用 Hooks 确保每一次 MCP 工具调用都在可控的边界内执行。不要在 MCP 服务端做权限控制，让 Hooks 统一承担这个职责——这样权限策略对所有工具（内置 + MCP）一视同仁。</p>
