## 笔记整理

<p>Agent Teams 是 Claude Code 的实验性功能，允许多个 AI Agent 在网状拓扑中直接通信、互相挑 战、自我协调。与 Sub-agents 的&quot;汇报制&quot;不同，Agent Teams 更像一支真正的团队：成员之间可以 直接对话，共享任务列表，Lead 负责综合协调。</p>

<p>两种架构对比：Hub-and-spoke vs Mesh Sub-agents 和 Agent Teams 代表了两种根本不同的 Agent 协作哲学：</p>

<p>不推荐 推荐 Sub-agents：Hub-and-spoke（汇报制） Agent Teams：Mesh（网状直连） · 主会话作为中心枢纽（Hub） · Teammates 之间可以直接互发消息 · 子代理只能向主 Agent 单向汇报 · 共享任务列表，自主认领协调 · 子代理之间无法直接通信 · 支持&quot;辩论&quot;收敛复杂问题 · Context 完全隔离，结果精简返回 · 各自独立 context，但通信频繁 · 稳定可用，Token 成本较低 · 实验性，Token 成本随成员线性增长 · 最佳场景：专注任务、结果导向 · 最佳场景：复杂调研、需要辩论协商 Agent Teams 默认禁用，需要在配置中显式开启：</p>

<p>启动 Agent Team 的提示词写法</p>

<p>1 | 基础启动：直接描述团队任务</p>

<p>创建一个 agent 团队调查线上 bug：用户登录后应用崩溃。</p>

<p>生成 3 个 teammate：</p>

<p>- auth-investigator：调查认证流程 - session-investigator：调查会话管理 - devil-advocate：专门寻找反驳其他两人理论的证据 让他们互相讨论直到达成共识，然后汇报根因。</p>

<p>2 | 并行代码审查（指定模型和数量）</p>

<p>创建一个 4 人团队并行重构这些模块，每个 teammate 使用 Sonnet 模型：</p>

<p>- frontend-refactor：重构 UI 组件层 - api-refactor：重构 API 接口层 - service-refactor：重构业务逻辑层 - test-refactor：更新对应测试用例</p>

<p>3 | 要求计划审批（高风险任务）</p>

<p>生成一个 architect teammate 重构认证模块。</p>

<p>要求他在开始实施前提交完整的重构计划供我审批， 审批通过后再开始修改代码。</p>

<p>SendMessage 通信机制 Agent Teams 中的通信通过消息系统（Mailbox）实现。Lead 可以向特定 Teammate 发消息，Teammates 之 间也可以直接通信：</p>

<p>消息类型 | 发送方式 | 适用场景 | 成本</p>

<p>message 点对点，发给指定 teammate 分配任务、请求结果、具体沟通低 broadcast 广播给所有 teammates 团队级通知、全员上下文同步高，谨慎使用</p>

<p>空闲通知 | Teammate 完成时自动触发 | Lead 据此综合结果 | 无额外开销</p>

<p>注意 broadcast 消息会被发送给所有 teammates，成本随团队规模线性增长。除非需要全员同步关键信息，否则优 先使用点对点 message。</p>
