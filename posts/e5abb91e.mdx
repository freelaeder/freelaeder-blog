## 笔记整理

<p>Sub-agents 是 Claude Code 多 Agent 系统的核心单元：每个子代理拥有独立的 system prompt、独 立的工具权限、独立的 context 窗口，能够专注完成特定任务并将结果返回主会话。理解 Sub-agents 的运行方式，是构建高效 Agent 工作流的第一步。</p>

<p>什么是 Sub-agents 在传统的 Claude Code 会话中，所有工作都在同一个 context 窗口中累积。随着任务复杂度上升，context 很 快被日志、中间结果、调试输出填满，主会话陷入&quot;噪声&quot;。Sub-agents 的出现改变了这个局面：它将特定的工 作单元委托给一个独立的 AI 实例，让主会话保持精简。</p>

<p>核心思路：Sub-agents 是&quot;摘要返回&quot;而非&quot;转录返回&quot;。子代理可以在独立的 200k token 空间中处理复杂任 务，但只把关键结论带回主会话——主会话的 context 得以保持清洁。</p>

<p>五个内置子代理 Claude Code 开箱即带五个内置子代理，覆盖最常见的专属任务：</p>

<p>代理名称 | 模型 | 工具权限 | 典型触发场景</p>

<p>Explore Haiku（低延迟） 只读工具文件搜索、代码探索、快速定位 Plan 继承主会话只读工具 Plan 模式下的上下文收集阶段 General-purpose 继承主会话全部工具复杂多步骤任务 Bash 继承主会话 Bash 在独立 context 执行 Shell 命令 Claude Code Guide Haiku — 回答 Claude Code 功能相关问题 注意 重要限制：子代理不能再生成子代理。如需嵌套委托，应使用 Skills 或从主会话顺序链接多个子代理调用。</p>

<p>子代理文件格式 自定义子代理以 YAML frontmatter + Markdown 正文的格式定义，存放在 .claude/agents/ 目录（项目级） 或 ~/.claude/agents/ （用户级全局）：</p>

<p>--- description: 深度调研专家。在需要调研特定技术主题时主动使用。</p>

<p>--- 你是一名专业的技术调研专家。</p>

<p>调研流程：</p>

<p>1. 明确调研目标和范围 2. 搜索官方文档和权威资料 3. 整理关键发现和技术细节 4. 输出结构化调研报告 输出格式：</p>

<p>- 以 Markdown 格式输出 - 包含具体的代码示例 - 标注信息来源 15 个 Frontmatter 字段完整说明 子代理的全部行为由 frontmatter 控制。以下是 15 个字段的完整参考：</p>

<p>必 字段</p>

<p>取值示例 | 说明</p>

<p>填 name 是 code-reviewer 唯一标识，小写字母 + 连字符，用于 @-mention 调用 description 是触发时机描述 Claude 据此判断何时委托，描述越清晰委托越准确 tools 否 Read, Grep, Bash 允许使用的工具白名单；缺省时继承主会话权限 disallowedTools 否 Write, Edit 禁止使用的工具黑名单；与 tools 互斥使用 model 否 sonnet / opus / 指定模型；inherit 表示继承主会话模型 permissionMode 否 acceptEdits default / acceptEdits / dontAsk / maxTurns 否 50 最大轮次限制，防止子代理无限运行 skills 否 [article-style] 启动时注入的 Skill 内容列表 mcpServers 否服务器配置仅对该子代理生效的 MCP 服务器 hooks 否生命周期钩子子代理的 PreToolUse / PostToolUse 钩子配置 memory 否 project 持久记忆范围：user （跨项目）/ project （项目级）/ local （本地不提交） background 否 true 设为 true 时始终以后台模式运行 effort 否 high 思考深度：low / medium / high / max （仅 Opus 4.6 支持） isolation 否 worktree 设为 worktree 时在独立 git worktree 中运行，适合并行 写文件任务 agent 否 code-reviewer 写入 settings.json 可将此子代理设为默认启动代理 存储位置与优先级 优先级 1（最高）：--agents CLI 参数（仅当前会话有效） 优先级 2：.claude/agents/（项目级，建议检入版本控制） 优先级 3：~/.claude/agents/（用户级，全局可用） 优先级 4（最低）：插件的 agents/ 目录 同名子代理遵循&quot;高优先级覆盖低优先级&quot;原则，项目级定义会覆盖用户级的同名代理。</p>

<p>三种调用方式</p>

<p>1 | 自然语言委托（自动判断）</p>

<p>Claude 根据代理的 description 字段自行判断是否委托。适合日常使用。</p>

<p>使用 test-runner 子代理修复所有失败的测试</p>

<p>2 | @-mention（强制指定）</p>

<p>在输入框输入 @&quot;agent-name (agent)&quot;，强制使用指定子代理，无需 Claude 自行判断。</p>

<p>@&quot;code-reviewer (agent)&quot; 审查 auth 模块的最新改动</p>

<p>3 | CLI 启动（整个会话作为子代理）</p>

<p>以指定代理身份启动整个 Claude Code 会话，适合自动化脚本。</p>

<p># 或通过 settings.json 设置默认 前台 vs 后台运行 子代理有两种运行模式，适合不同场景：</p>

<p>不推荐 推荐 后台子代理（background: true） 前台子代理（默认） · 阻塞主会话直到完成 · 与主会话并发运行，不阻塞 · 权限提示直接透传给用户 · 启动前须预审批工具权限 · 可使用 AskUserQuestion 工具 · 需要澄清时工具调用失败但继续 · 适合需要人工确认的关键操作 · 适合长时调研、批量处理任务 用户可随时按 Ctrl+B 将进行中的前台任务转为后台运行，继续处理其他工作。</p>

<p>核心建议 最小权限原则：在 tools 字段中只赋予子代理必要的工具。调研类子代理只需 Read/Grep/WebFetch，无需 Write/Edit；实现类子代理无需 WebSearch。权限越小，风险越低，Claude 的聚焦度越高。</p>
