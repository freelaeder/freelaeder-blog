## 笔记整理

<p>当多个 Agent 并行写入同一代码库时，文件冲突是最大的隐患。Git Worktree 隔离为每个 Agent 提供 完整仓库的独立副本，彻底消除并行写冲突，同时保留完整的 git 历史追踪能力。本节介绍 Worktree 隔离的配置方式、自动清理机制和三大典型应用场景。</p>

<p>为什么需要 Worktree 隔离 不推荐 推荐 Worktree 隔离后 不隔离时的问题 · 并行 Agent 写入同一文件 → 静默覆盖冲突 · 每个 Agent 拥有完整仓库的独立副本 · Agent A 的中间状态影响 Agent B 的判断 · 各自独立的工作目录 + 独立的 git 分支 · 无法回滚单个 Agent 的工作 · 无改动时自动清理，有改动时保留审查 · 合并时需要手工解决大量冲突 · 逐个合并 PR，历史清晰可追溯 --worktree 标志的三种用法</p>

<p>1 | 指定名称创建 worktree</p>

<p>每个 worktree 对应一个独立的 git 分支（worktree-feature-auth ），在独立目录中完整克隆仓库状态。</p>

<p>2 | 自动生成名称</p>

<p>适合临时性探索任务，无需事先规划名称。</p>

<p>3 | 并行启动多个独立会话</p>

<p># 三个会话并行运行，各自在独立目录中工作 子代理 Worktree 隔离配置 在子代理的 frontmatter 中设置 isolation: worktree，每次调用该子代理时都会自动获得独立的 worktree：</p>

<p>--- description: 并行实现专家，适合批量代码迁移、多模块同步重构任务 --- 你是专注的实现专家。在独立的 worktree 中完成指定任务， 不干扰其他并行 Agent 的工作。完成后提交到当前分支， 主会话将整合各分支的改动。</p>

<p>自动清理机制 Claude Code 对 worktree 的生命周期管理遵循&quot;有改动才保留&quot;的原则：</p>

<p>Agent 退出 检测是否 无改动：</p>

<p>worktree 会话 | →</p>

<p>有代码改动 | →</p>

<p>自动删除 有改动：</p>

<p>→ 保留等待审查 手动管理 worktree：</p>

<p># 建议在 .gitignore 中添加：</p>

<pre><code>.claude/worktrees/ # 防止 worktree 内容污染主仓库 git status</code></pre>

<p>Worktree 目录结构</p>

<pre><code>├── feature-auth/ # 对应分支：worktree-feature-auth</code></pre>

<pre><code>│ ├── src/ # 完整仓库副本</code></pre>

<pre><code>│ └── .git # git worktree 引用文件</code></pre>

<pre><code>├── bugfix-123/ # 对应分支：worktree-bugfix-123</code></pre>

<pre><code>│ └── ...</code></pre>

<pre><code>└── bright-running-fox/ # 自动命名的临时 worktree</code></pre>

<pre><code>└── ...</code></pre>

<p>三大典型应用场景 场景一：批量代码迁移 将大量文件从旧 API 迁移到新 API，是最适合 Worktree 并行的任务类型。</p>

<p>1 | 分析迁移范围</p>

<p>先用主会话列出所有需要迁移的文件，按数量均分给多个子代理。</p>

<p>2 | 并行分组迁移</p>

<p>需要将 50 个文件从 API v1 迁移到 API v2：</p>

<p>- Agent 1（worktree-batch-1）：处理文件 01-10 - Agent 2（worktree-batch-2）：处理文件 11-20 - Agent 3（worktree-batch-3）：处理文件 21-30 - Agent 4（worktree-batch-4）：处理文件 31-40 - Agent 5（worktree-batch-5）：处理文件 41-50 全部并行在独立 worktree 中运行。</p>

<p>3 | 逐批合并 PR</p>

<p>各 worktree 完成后，分别创建 PR，逐个 review 合并，历史清晰互不干扰。</p>

<p>场景二：A/B 方案探索 面对重要技术决策时，让两个 Agent 分别在独立 worktree 中实现两种方案，最后人工对比选择。</p>

<p># 方案 A：使用 JWT 实现认证 # → Agent A 在 worktree-approach-jwt 中实现完整的 JWT 方案 # 方案 B：使用 Session + Redis 实现认证 # 两个会话并行运行，完成后人工对比代码质量、性能、复杂度 A/B 方案的最大价值：不需要先做选择再实施，可以同时得到两个完整可运行的实现，基于真实代码而不是理 论分析来做技术选型，决策质量大幅提升。</p>

<p>场景三：Feature + Bugfix 并行开发 线上有紧急 bug 需要修复，同时新功能开发不能中断——Worktree 让这两件事可以真正并行：</p>

<p># 会话 1：新功能开发（在独立 worktree 中，不受 bugfix 影响） # Agent 1 专注开发 OAuth 登录功能，完整的代码环境 # 会话 2：紧急 bugfix（在另一个独立 worktree 中） # Agent 2 专注修复 session 内存泄漏，不干扰功能开发 # 两个会话完成后，分别创建 PR，独立 review 合并 Worktree 与 Sub-agents 结合使用 将子代理的 isolation: worktree 配置与并行调用结合，可以实现最干净的并行架构：</p>

<p># 主会话发出指令，子代理各自在独立 worktree 中工作 并行启动三个 parallel-implementer 子代理：</p>

<p>子代理 1：迁移 src/components/ 目录下的所有组件到新的 Design System API 子代理 2：迁移 src/hooks/ 目录下的所有 hooks 到新的状态管理方案 子代理 3：更新 tests/ 目录下对应的测试用例 每个子代理完成后输出改动摘要，我来审查并合并。</p>

<p>核心建议 最佳实践总结：领域拆分 + Worktree 隔离 + 独立 PR 是并行开发的黄金组合。领域拆分确保文件无交集， Worktree 提供物理隔离，独立 PR 保证历史清晰可追溯。三者缺一不可。</p>
