## 笔记整理

<p>Claude Code 是天生的 Git 原住民。从生成 commit message、创建 PR，到双会话 Code Review 模 式，它能将 Git 工作流中最耗时的环节全部自动化，同时保持可审查性和安全性。</p>

<p>Git 集成的基础能力 Claude Code 原生集成了 git 命令和 gh CLI，理解 GitHub 工作流的全部概念。你可以用自然语言描述 Git 操 作，Claude 会正确翻译并执行：</p>

<p>自然语言指令 | Claude 执行的命令</p>

<p>&quot;查看这次改动了哪些文件&quot; git status &amp;&amp; git diff --stat &quot;这次改动和 main 的差异&quot; git diff main...HEAD &quot;最近 5 个 commit 做了什么&quot; git log --oneline -5 &quot;找到引入这个 bug 的 commit&quot; git log -p --follow -S &quot;buggy_function&quot; &quot;在新分支上开始这个功能&quot; git checkout -b feat/feature-name 手写 commit message 是开发者最容易&quot;偷懒&quot;的环节，结果是项目历史变成&quot;fix bug&quot;、&quot;update&quot;、&quot;test&quot;的乱 账。Claude Code 可以根据实际改动生成规范、有意义的 commit message。</p>

<p>基础用法 检查 git diff，生成一条描述性的 commit message。</p>

<p>指定规范格式 - feat: 新功能 - fix: bug 修复 - docs: 文档更新 - refactor: 重构（不影响功能） - test: 测试相关 - chore: 构建/工具链 格式：type(scope): 简短描述（中文） 正文（可选）：</p>

<p>说明&quot;为什么&quot;而不只是&quot;做了什么&quot;。</p>

<p>含正文的完整 Commit 提交这次改动，commit message 需要：</p>

<p>1. 第一行：type(scope): 描述（50字以内） 2. 空一行 3. 正文：解释为什么做这个改动，而不只是列出改了什么 4. 如果有 breaking change，在正文结尾加 BREAKING CHANGE: 说明 在 CLAUDE.md 中固化 Commit 规范 # Git 规范 - 格式：type(scope): 描述（中文，祈使句） - 正文必须说明&quot;为什么&quot;而不只是&quot;做了什么&quot; - Breaking change 必须在正文中标注 - 不使用 --no-verify 跳过 hook - 不使用 --force 推送 main 分支 Claude Code 通过 gh pr create 创建 PR，可以自动总结改动、填充描述模板，并处理整个推送流程。</p>

<p>基础 PR 创建 将当前分支推送到远程，创建 PR，目标分支是 main。</p>

<p>PR 标题用一行描述核心改动，描述包含改动摘要。</p>

<p>完整 PR 创建（含描述模板） 创建 PR，PR 描述使用以下模板：</p>

<p>## 改动摘要 （3 点以内，说明做了什么） ## 动机 （为什么需要这个改动？解决了什么问题？） ## 测试计划 （如何验证改动正确？列出测试步骤或测试用例） ## 风险与注意事项 （可能影响的模块、需要关注的边界情况） ## 相关链接 （关联的 Issue、设计稿、文档链接） 从 Issue 创建 PR 使用 &#x27;gh issue view 1234&#x27; 查看 issue 详情， 理解问题，在代码库中找到相关文件， 实现修复，写测试验证， 确保代码通过 lint 和类型检查， 创建带描述性 commit message 的 PR， 在 PR 描述中关联 closes #1234。</p>

<p>并行工作树（Git Worktrees） 当你需要同时处理多个任务（如主线开发 + 紧急 Bug 修复）时，Git Worktrees 允许在同一个代码库的不同工 作目录并行启动多个 Claude 会话，互不干扰：</p>

<p># 在独立工作树上开发新功能 # 在另一个工作树上修复紧急 bug（同一窗口的另一个标签页） 每个工作树对应一个独立的 git branch，修改完全隔离。完成后正常合并回主分支即可。</p>

<p>核心建议 并行工作树特别适合&quot;等待 CI/CD 构建期间做其他事&quot;的场景。在功能开发的 Session 等待测试运行时，在另一 个工作树里同步处理 Issue 修复，零等待时间。</p>
